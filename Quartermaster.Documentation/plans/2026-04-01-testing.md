# Testing Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add unit tests for the 6 critical business logic areas not yet covered by the existing 201 validator/sanitization tests.

**Architecture:** All tests that need database access require a test database setup. Two approaches: (A) mock repositories using interfaces, or (B) use an in-memory SQLite or real MySQL test database. Since the repositories use LinqToDB directly (not interfaces), option B is more practical — use a test MySQL database with the migration runner. For pure logic tests (like chapter resolution), in-memory data structures work.

**Tech Stack:** TUnit, existing test project, LinqToDB, FluentMigrator

## Test Plan: 6 Test Suites, ~64 Test Cases

### Task 1: Test Infrastructure — TestDatabaseFixture

Before writing business logic tests, create a shared test fixture that:
1. Creates a test MySQL database (`quartermaster_test`)
2. Runs FluentMigrator to create schema
3. Provides a fresh `DbContext` per test (with transaction rollback or truncation between tests)
4. Cleans up after all tests

This avoids mocking the data layer and tests the actual SQL queries.

**Files:**
- Create: `Quartermaster.Server.Tests/Infrastructure/TestDatabaseFixture.cs`

### Task 2: ChapterRepository Tests (~14 cases)

**File:** `Quartermaster.Server.Tests/Chapters/ChapterRepositoryTests.cs`

**GetAncestorChain:**
- [ ] Single node (no parent) → returns just itself
- [ ] Three-level hierarchy → returns [child, parent, grandparent]
- [ ] Circular reference protection (node with ParentChapterId == own Id) → doesn't loop forever
- [ ] Chain order is child-to-root

**GetDescendantIds:**
- [ ] Leaf node → returns just its own ID
- [ ] Two-level tree → returns parent + child IDs
- [ ] Multi-level tree → returns all descendant IDs
- [ ] Self-referencing protection

**FindByExternalCodeAndParent:**
- [ ] Exact match on code + parent → returns chapter
- [ ] Code match, wrong parent → returns null
- [ ] Parent match, wrong code → returns null
- [ ] Null parent (root chapter match) → returns chapter
- [ ] Non-existent code → returns null

**Seed data:** Create a 3-level chapter hierarchy (Bund → LV → Bezirk → Kreis) with ExternalCodes.

## Execution Notes

- Tasks 2-4 are the highest value — they test the core business logic that's hard to verify manually
- Task 5 (import) provides the most protection against regressions since imports run daily
- Task 1 (test infrastructure) is prerequisite for all others
- Tests should use a separate `quartermaster_test` database to avoid interfering with dev data
- Each test should be independent — seed its own data and clean up after
